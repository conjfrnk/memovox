# M0.1 — Observability & metrics spine

> **Wave:** 0 · **Effort:** L · **Status:** ✅ done (branch `phase4-observability`, 8/8 workstreams; 283 pass / 2 skip; 4 gates green)
> **Depends on:** none · **Owns (single-owner concerns):** the single structured-logging hook (stderr); the cap-surfacing edit to `loom/consolidate.py` `find_contradictions` (so the Scale track M0.2 does not re-edit the same site) · **Blocks:** all later tracks benefit — build FIRST
> **Spec:** §7 (Observability row: "structured logs + per-stage metrics + traces"), §9 (per-video token/compute budgets are configurable and logged; throughput; graceful degradation)

## Goal
Build the measurement spine the entire Phase-4 roadmap leans on: a stdlib `time.perf_counter` tracer that records, per pipeline stage, a span of `{wall_ms, status, counters, caps}`; a single structured-JSON logging hook that writes **only to stderr** (stdout is reserved for MCP JSON-RPC and CLI human output); a per-video token/compute **Budget** (soft by default, raising `BudgetExceeded` only in explicit hard mode); the conversion of every currently-silent cap (consolidation `max_claims`, retrieve `pool`/`top_k`, Tessera `frame_max`/`per_scene_cap`) from a bare list slice into a structured, counted **cap event**; a `metrics` dict attached to `IngestReport`/`Answer`/`ConsolidationReport`; persistence of per-video `stage_metrics` plus a cumulative ledger in the store; a new `memovox metrics` command and an extended `memovox stats`; and an optional OpenTelemetry exporter behind a `[otel]` extra that is a clean no-op when absent. Everything ships with a deterministic free-path default, and the eval harness gains an `observability` block that proves every stage emits a well-formed span and that a forced-small budget surfaces a cap event — without ever thresholding wall-clock magnitudes.

## Why it matters
Spec §9 promises measurable throughput, configurable-and-logged per-video budgets, and graceful degradation; the §7 stack names "structured logs + per-stage metrics + traces" as the Observability layer; the roadmap's literal Phase-4 "dashboards" bullet is this track surfaced (plus M3.4). Today none of it exists — there is no `logging`, no `perf_counter`, no per-stage timing anywhere in `src/` (every emit is a `print()` to stdout in `cli.py`, see `src/memovox/cli.py:35-208`). Without this spine, every later track's latency/throughput/recall claim (`M0.2` p95/recall, `M-X.1` visual parallelism, `M3.3` serving) is unfalsifiable, and the silent caps (`find_contradictions max_claims=600`, retrieve pool, frame caps) can quietly drop knowledge with no signal. Building it first also lets this track **own** the two collision-prone edit sites — the single stderr logging hook and the `consolidate.py` cap site — so M0.2 (Scale) inherits a reported cap instead of re-touching the same lines.

## Free-path guarantee
- **Defaults OFF / soft:** `metrics_enabled` defaults to a state that produces metrics *without changing any existing output bytes* — the tracer is always-on but cheap (a `perf_counter` pair + a dict per stage), and its only side effect is the new `metrics` dict on reports + stderr log lines + new store tables. None of that touches stdout, retrieval ordering, claim text, or any field the existing 247 tests/4 gates read. `budget_mode` defaults to `"soft"` (record overage, never raise). `otel_enabled` defaults OFF (no-op exporter).
- **Caps surface, behavior unchanged:** converting a silent slice into a cap *event* must not change *what* is kept. `[:max_claims]` still keeps the same first 600 claims; it just additionally records `caps: [{name:"max_claims", limit:600, dropped:N}]`. The retrieve `pool`/`top_k` and Tessera caps are reported identically. **Byte-identical guarantee:** retrieval result ids+order, answer text, citation list, contradiction pairs, keyframe selection, and the persisted digest stay exactly as today.
- **Fallback is the stdlib itself:** the tracer, logger, and budget are pure-stdlib (`time`, `logging`, `json`); there is no model or network dependency. OTel is the only optional piece and degrades to a no-op object with the same method surface when `opentelemetry` is not importable (same `try/import → fallback` idiom as `src/memovox/backends/`).
- **Must stay byte-identical** (asserted): the JSON of `Answer.to_dict()` **excluding** the new `metrics` key; retrieval top-k ids and order; `find_contradictions` pair set; `select_keyframes` output indices; the per-video digest file. The frozen eval-settings snapshot records the new flags' default values so a future flip is caught.
- **Determinism of the metrics themselves:** counters and cap counts are deterministic; `wall_ms`/timestamps are NOT (machine-dependent), so they are explicitly excluded from every equality/threshold assertion. The logger uses a fixed, sorted key order and `ensure_ascii=False` to keep emitted lines stable in structure.

## Risks & mitigations
- **Byte-identity drift from cap-surfacing (provenance is sacred).** Converting a slice to a counted event could accidentally change *what* is kept. → W3's Red tests assert the returned pair set / top-k ids+order / kept-index list are byte-identical to a pre-change baseline; only the *event presence* is the new assertion.
- **Stray output on stdout breaks MCP JSON-RPC** (review risk 5). → Single logging hook hard-wired to `sys.stderr` (asserted `handler.stream is sys.stderr`); a test captures stdout during ingest/ask/consolidate/MCP and asserts no JSON log line leaks. No `print()` added anywhere in the spine.
- **Determinism erosion from new default-OFF flags** (review risk 2). → Frozen eval-settings snapshot pins `budget_mode`/`otel_enabled`/`metrics_enabled`; `wall_ms`/timestamps excluded from all equality/threshold assertions; counters/cap counts are deterministic.
- **`metrics` key perturbs report-equality tests / downstream consumers.** → New key is additive; byte-identity guards compare `to_dict()` *minus* `metrics`. The harness never reads it.
- **Schema migration on existing stores.** → New tables added via `_SCHEMA` `CREATE TABLE IF NOT EXISTS` + idempotent `_migrate` (mirrors the existing `ALTER TABLE … ADD COLUMN` guard at `src/memovox/loom/store.py:155-158`); `SCHEMA_VERSION` bump is forward-only and the ledger upsert is conflict-safe.
- **Per-stage timing overhead.** → `perf_counter` + a dict per stage is negligible vs. ASR/embed; the tracer does no I/O on the hot path (logging is one formatted line per span). No magnitude is gated, so jitter can't flake CI.
- **OTel import surface (free-path purity).** → Guarded `try/import` with a `_NullOtelSpan` fallback; `[otel]` is never installed in CI; the no-op path is the tested default.
- **Collision with M0.2 on `consolidate.py`** (the reason this is single-owner). → This track lands the cap *event* on `ConsolidationReport.metrics`; M0.2 consumes it for "reported paging" and does not re-edit lines 56-67.
- **`frame_max` config/function-default mismatch** (`Settings.frame_max=1200` vs `sample_frame_signatures(max_frames=600)`). → Surface both in the cap counter so the discrepancy is visible; flag it as an open question rather than silently "fixing" it here (a fix could change visual output and belongs in M1.1/M-X.1).
