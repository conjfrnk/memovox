# M3.3 — Serving & deployment

> **Wave:** 3 · **Effort:** L · **Status:** not started
> **Depends on:** M0.2 (incremental consolidation — `consolidate(..., since_watermark=…)` primitive, the single owner) · M3.2 (the `Memovox.sync()` rewrite — the single owner) · **Owns (single-owner concerns):** none · **Blocks:** none
> **Spec:** §7 (named backends behind a common interface — here FastAPI as an optional serving backend), §8 (the four interfaces: CLI / SDK / REST / MCP), §9 (throughput / scale / observability / graceful degradation), §12 (risks — ToS / private-by-default posture)

## Goal
Make memovox's long-running operations (`consolidate`, `sync`, `ingest`) run **async, resumable, and off the caller thread** without taking on a web framework or breaking the stdlib-only default. This track adds a stdlib `threading` + SQLite-backed **job runner** (`queued → running → succeeded|failed`, attempt count, exponential-backoff retry, idempotent `(kind, args_hash)` de-dup, persisted in a `jobs` table), makes the MCP `consolidate` tool **non-blocking** (returns a `job_id`) and adds a `job_status` tool, extracts the existing REST handler bodies into shared `server/routes.py` **pure functions** so an **optional FastAPI app behind `[serve]`** mounts the *same* routes as the stdlib `http.server` (a JSON-parity test guards them), ships a `memovox-worker` console script + a `Dockerfile` + `docs/DEPLOY.md` documenting free / serve / worker modes, and folds in the §12/§9 **private-by-default posture** (a `local_only` egress toggle plus the policy surface). Default worker concurrency is **1 and deterministic**; >1 is opt-in and out of the eval gate.

## Why it matters
Spec §8 promises four co-equal interfaces (CLI, SDK, REST, MCP) and §9 calls consolidation a *background* job — but today every one of these calls runs **synchronously on the caller's thread**. `Memovox.consolidate()` (`src/memovox/sdk.py:78`) blocks until the whole cross-corpus pass finishes; the MCP `consolidate` tool (`src/memovox/server/mcp.py:155`) blocks the JSON-RPC loop; `sync` (`src/memovox/sdk.py:43`) downloads and ingests inline. For an agent sitting in Claude Code, "consolidate the corpus" should return a handle immediately and let the agent keep working, not freeze the editor for a minute. This track delivers that resumability and async posture using only `threading` + SQLite (no Celery, no Redis, no framework), keeping the free path intact. It also (a) lets a real deployment run a `memovox-worker` next to the REST API, (b) makes the optional FastAPI production server documented in `src/memovox/server/rest.py:2` actually exist and provably return identical JSON to the free server, and (c) addresses the §12 risk that a knowledge engine ingesting other people's videos must be **private-by-default** and respect source terms — beyond the single `local_only` flag, a stated posture.

## Scope (reconciled)
In scope:
- **Stdlib job runner (`src/memovox/serving/jobs.py`)** — a `jobs` table (`job_id`, `kind`, `args_hash`, `args_json`, `state`, `attempts`, `max_attempts`, `next_run_at`, `result_json`, `error`, `created_at`, `updated_at`), states `queued|running|succeeded|failed`, attempt count + **exponential backoff** retry, and **idempotent `(kind, args_hash)` de-dup** (enqueueing an identical not-yet-terminal job returns the existing `job_id`). Pure SQLite + `threading`; no new dependency.
- **Job worker (`JobWorker`)** — a `threading.Thread` loop that claims the next `queued` job whose `next_run_at` has passed, runs it, records `succeeded`/`failed` + backoff, and persists every transition so a crash is resumable on restart. **Default concurrency = 1** (deterministic, serial). The job kinds dispatch to `Memovox.consolidate` / `Memovox.sync` / `Memovox.ingest`.
- **Non-blocking MCP `consolidate` + `job_status` tool** — `consolidate` enqueues a job and returns `{job_id, state}` immediately instead of blocking; a new `job_status` MCP tool resolves a `job_id` to its state/result/error. The synchronous SDK/CLI `consolidate` is preserved (so M0.2's `incremental_equivalence` gate and the eval harness, which call it inline, are unchanged).
- **`server/routes.py` pure functions** — extract the request-handling bodies currently inlined in `make_handler` (`src/memovox/server/rest.py:27`) into framework-agnostic pure functions (`route_ingest(mv, body) -> (status, dict)`, `route_query`, `route_clip(mv, params)`, `route_export`, `route_videos`, `route_contradictions`, `route_synthesize`, plus `route_consolidate` / `route_job_status`). The stdlib `http.server` handler and the optional FastAPI app both call these — single source of truth for request semantics.
- **Optional FastAPI app behind `[serve]` (`src/memovox/server/fastapi_app.py`)** — `is_available()`-gated, imports `fastapi` only when installed; mounts the same `routes.py` functions. A **parity test** asserts the stdlib server and FastAPI app return byte-identical JSON for the same inputs.
- **`memovox-worker` console script + `worker` CLI subcommand** — a `[project.scripts]` entry (`memovox-worker = "memovox.cli:worker_main"`) and a `memovox worker` subcommand that runs the `JobWorker` loop (stderr logging, `--concurrency`, `--once`/drain mode for tests).
- **`Dockerfile` + `docs/DEPLOY.md`** — documenting the three modes: **free** (stdlib `memovox serve` + inline jobs), **serve** (`pip install memovox[serve]` → FastAPI + uvicorn), **worker** (`memovox-worker` next to the API, sharing the store).
- **Private-by-default posture (folded in from review, §12/§9)** — a `local_only` egress toggle in `Settings` (when on, URL acquisition / any network model fetch is refused with a clear error) **plus** the stated policy in `docs/DEPLOY.md`: private-by-default store, respect source terms-of-service, and a retention/redaction note (the `delete_video` path at `src/memovox/loom/store.py:215` is the redaction primitive).
- **Eval** — `serving.equivalent`: background-enqueued + drained consolidation produces the **same counts** as an inline `Memovox.consolidate()`; plus the stdlib==FastAPI JSON parity test. Both land **ungated-then-gated** as exact-equivalence invariants.

Folded in from the completeness review:
- The §12/§9 **ToS / private-by-default posture beyond a bare `local_only` flag** — the flag *and* a documented policy surface (private store, source-ToS respect, retention/redaction).
- The **honesty constraint:** default worker concurrency = 1 (deterministic); >1 is opt-in and **out of the eval gate**. The real throughput win is frame parallelism (M-X.1 in the program roadmap), which this track does not deliver and must not claim.
- The **frozen eval-settings-snapshot discipline:** every new default-OFF flag this track adds (`local_only`, worker `concurrency`, FastAPI selector) is pinned in M0.2's frozen `_EVAL_SETTINGS_SNAPSHOT`, not just `_FREE_BACKENDS`.

Non-goals / deferrals:
- **The incremental-consolidation watermark/scoping is NOT reimplemented here.** It is owned by M0.2; this track's job runner *calls* `consolidate(store, *, nli, since_watermark=…)` (the M0.2 primitive). Re-deriving the watermark is forbidden (the review's single-owner reconciliation).
- **The `Memovox.sync()` rewrite is NOT in scope.** It is owned by M3.2; the `sync` job kind dispatches to whatever `Memovox.sync()` M3.2 ships. This track only schedules it.
- **Multi-worker / concurrent throughput is opt-in and ungated.** `concurrency > 1` is supported but not the default, not deterministic across interleavings, and never in the CI gate.
- **FastAPI/uvicorn are an optional upgrade, never required.** The free path is the stdlib `http.server` + inline or threaded jobs; FastAPI is gated by `[serve]` install and `is_available()`.
- **No external queue/broker** (Celery/Redis/RabbitMQ). The runner is SQLite + `threading` only.

## Free-path guarantee
- **Zero new runtime dependencies on the default path.** The job runner is `threading` + `sqlite3` + `hashlib` (for `args_hash`) + `json` — all stdlib. FastAPI/uvicorn are imported **only** inside `server/fastapi_app.py` behind an `is_available()` probe and the `[serve]` extra; the free server stays the stdlib `http.server` (`src/memovox/server/rest.py`).
- **The synchronous `consolidate` stays byte-identical.** `Memovox.consolidate()` (`src/memovox/sdk.py:78`) is **not** changed to go through the queue — it remains the inline, deterministic call the eval harness depends on (`eval/harness.py:520`). The job runner is an *additional* async path (`Memovox.enqueue_consolidate()` / the MCP tool), and a *drained* background consolidate must produce the **same counts** as the inline call (the `serving.equivalent` gate). The free stack is pinned by M0.2's frozen snapshot.
- **`routes.py` extraction is a pure no-op refactor.** Moving the handler bodies into pure functions must not change a single byte of REST response. A parity-style assertion (the stdlib handler before/after returns identical JSON for the same request) locks this, mirroring M0.2's no-op-refactor discipline.
- **Everything new defaults OFF / serial.** Worker `concurrency` defaults to `1` (deterministic serial execution — identical to running jobs one at a time). `local_only` defaults `False` (today's behavior — URL acquisition allowed). FastAPI selector defaults to the stdlib server. Each is added to M0.2's frozen `_EVAL_SETTINGS_SNAPSHOT` so a future default flip cannot silently move a gate number.
- **MCP stdout discipline.** All worker and job-runner logging goes to **stderr** (never stdout — MCP's JSON-RPC transport owns stdout). No stray `print` in `serving/` or the worker. The existing `print` in `rest.serve()` (`:138`) is acceptable (REST server, not MCP) but the worker entry point logs to stderr.
- **Idempotency preserved.** `(kind, args_hash)` de-dup means re-enqueueing the same consolidate while one is pending returns the same `job_id` and does no extra work — consistent with the deterministic/idempotent core. The drained result equals the inline result.

## Eval gate
Two new equivalence checks; the existing four must stay green:
- **`serving.equivalent == 1.0`** — a background-enqueued, worker-drained consolidation produces the **same `ConsolidationReport` counts** as an inline `Memovox.consolidate()` on the identically-ingested golden corpus. Lands **ungated** in W8 (reported), then **gated at `1.0`**. It is an exact-equivalence correctness invariant (not a statistical metric), so per the global discipline it gates immediately rather than waiting for ≥3 golden items — same rationale M0.2 used for `parity`/`incremental_equivalence`.
- **stdlib==FastAPI JSON parity** — a **test-level** gate (`tests/test_fastapi_parity.py`), not a harness number: FastAPI and the stdlib server return byte-identical JSON for every route (both call `routes.py`). It **skips cleanly** when `fastapi` is absent (so a bare machine's `make test` stays green with extra skips); it runs and must pass on a `[serve]`-installed machine.
- **Existing gates unchanged and confirmed green after every workstream:** `retrieval.hit_rate >= 0.6`, `groundedness >= 0.8`, `contradiction.f1 >= 0.5`, `synthesis.groundedness >= 0.8` (`eval/harness.py:642`–`649`). Because the synchronous `Memovox.consolidate()` / `mv.contradictions()` path the harness calls inline (`eval/harness.py:520`) is **deliberately left untouched**, these gates cannot move; verify explicitly after W4 (the most likely to regress them by accident).
- **Worker `concurrency > 1` is opt-in and NEVER gated.** Default `concurrency=1` is the only deterministic mode; the gate runs at `concurrency=1`. Multi-worker interleavings are out of scope for the eval and the honesty note in `DEPLOY.md` says so.

Per the global discipline: every new metric lands ungated until ≥3 stable golden items exist, **except** the exact-equivalence invariants above (`serving.equivalent`, the JSON parity test), which are correctness assertions on the existing corpus and gate immediately.

## Risks & mitigations
- **Threading + SQLite contention (review-flagged).** Multiple `LoomStore`/`JobStore` connections writing the same `memovox.db` can hit `database is locked`. Mitigate: **per-job own connection** (never share a connection across threads), **WAL** (already on at `store.py:136`) + an explicit **`PRAGMA busy_timeout`** on every connection (the worker's and `JobStore`'s — `store.py` currently sets WAL but not `busy_timeout`), and **serial-by-default** (`concurrency=1`). Concurrency >1 is opt-in and ungated, so the gate never exercises contention.
- **The async path drifting from the inline path.** A background consolidate could subtly diverge from inline (different settings, different store state). Mitigate: the `serving.equivalent` gate compares drained counts to inline counts; the job dispatcher calls the **same** `Memovox.consolidate()` / M0.2 `consolidate(..., since_watermark=…)` primitive, not a reimplementation.
- **MCP stdout pollution (review-flagged repo-wide).** A stray `print` in the worker or job runner corrupts the JSON-RPC stream. Mitigate: all `serving/` and worker logging to **stderr**; a grep-style assertion / review item that `src/memovox/serving/` contains no `print(` to stdout; the existing `rest.serve()` `print` (`:138`) is REST-only and left as-is.
- **Single-owner boundary leakage.** Temptation to re-derive the consolidation watermark or rewrite `sync` here. Mitigate: the `consolidate` job calls M0.2's `consolidate(..., since_watermark=…)`; the `sync` job calls M3.2's `Memovox.sync()` — both as-shipped. This doc's scope explicitly forbids reimplementation (the review's reconciliation #1/#2).
- **FastAPI optionality breaking the bare machine.** A top-level `import fastapi` anywhere on the default path would break the stdlib-only guarantee. Mitigate: `fastapi` imported only inside `server/fastapi_app.py` behind `is_available()`; the parity test **skips** when absent; `tests/test_fastapi_parity.py::test_fastapi_unavailable_is_clean` proves the unavailable path raises `BackendUnavailable`, never an `ImportError` crash.
- **Idempotency edge: re-enqueue after terminal state.** De-dup is on *non-terminal* `(kind, args_hash)`; a new enqueue after a job `succeeded` must create a fresh job (so a second consolidate can run). Mitigate: the `WHERE NOT EXISTS (… state IN ('queued','running'))` clause; a test asserts re-enqueue-after-success creates a new `job_id`.
- **Frozen-snapshot omission.** `local_only`, worker `concurrency`, FastAPI selector not pinned in M0.2's snapshot would let a future default flip silently move a gate. Mitigate: W7/W8 add them to `_EVAL_SETTINGS_SNAPSHOT` and (per M0.2's W6) a snapshot test asserts every new default-OFF flag is pinned.
- **Honesty: no real throughput win.** `concurrency=1` is deterministic but not faster; the visual track is the bottleneck (program roadmap M-X.1). Mitigate: `DEPLOY.md` states this explicitly; the eval gate does not claim or measure throughput.
- **`tests/test_mcp.py` tool-count assertion churn.** Adding `job_status` changes `len(names) == 6` (`:45`). Mitigate: W4 updates that assertion in the same commit as the tool addition (a known, contained blast radius).
