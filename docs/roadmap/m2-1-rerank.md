# M2.1 — Cross-encoder rerank stage

> **Wave:** 2 · **Effort:** M · **Status:** not started
> **Depends on:** none (coordinate with M2.2 planner and M2.3 clips — all three touch `augur/answer.ask`; build order is **rerank → planner → clips**) · **Owns (single-owner concerns):** none · **Blocks:** none
> **Spec:** §5 ("dense + sparse + graph … fuse with Reciprocal Rank Fusion; **rerank with a cross-encoder**"), §3 (the Augur diagram stage `plan → hybrid retrieve → **rerank** → grounded answer + timestamped clip citations`)

## Goal
Make the spec's `rerank` stage a first-class, swappable step that sits **between fused retrieval and answer synthesis**. Introduce a `Reranker` backend interface mirroring the existing `Embedder`/`NLIBackend`/`LLMBackend` registry pattern, with a deterministic, dependency-free free fallback and an optional cross-encoder upgrade behind `is_available()`. The default (free) path is a **proven non-regressing reorder** — by construction either identity or a reorder that cannot move the eval numbers — so every current retrieval/groundedness/synthesis gate stays byte-identical, while the upgrade slot lets a real cross-encoder lift `mrr`/`ndcg` on the golden corpus.

## Why it matters
Spec §5 and the §3 architecture diagram both name a `rerank` stage; today there is **no implementation** — `augur.answer.ask` jumps straight from RRF fusion to citation construction (`/Users/connor/projects/memovox/src/memovox/augur/answer.py:87-94`). RRF fuses by *rank position* across legs and is deliberately blind to the actual query↔candidate text match, so the empirically strongest video-RAG configuration (hybrid retrieval **+ reranking**) is currently only half-built. Landing the rerank seam now (a) closes the §5/§3 gap with a deterministic default that changes nothing, (b) gives M3.4's A/B benchmark a real upgrade slot to rank, and (c) unblocks a measurable retrieval-quality lever (cross-encoder relevance) without touching the free-stack guarantees. It is sequenced **first** in Wave 2 because the planner (M2.2) and clip stitching (M2.3) both build on top of the answer pipeline, and a stable rerank seam is the cleanest thing to lay down before they reshape `ask`.

## Scope (reconciled)
In scope:
- A `Reranker` ABC in `backends/base.py` mirroring the existing backend interfaces: a `score(query, candidates)` / `rerank(query, candidates) -> reordered` contract, an always-`True` `is_available()` on the free fallback, `name` attribute, and the standard `Backend.__init__(config, **options)` shape.
- A **deterministic free fallback** registered as the default. Two acceptable implementations (decide in W2; see Open questions): (a) **identity** reranker — returns candidates untouched (guaranteed byte-identical), or (b) a **lexical/cross-overlap reorder** that is *proven non-regressing* on golden (re-sorts only within the already-retrieved set using query↔moment content-token overlap, stable-sorted so ties never perturb today's order). The fallback MUST NOT add or drop candidates — it only reorders the fused set.
- A `get_reranker(name="auto", *, config, **options)` factory in `backends/__init__.py` following the `get_nli`/`get_embedder` precedent (`"none"`/`""` → identity fallback; `"auto"` → cross-encoder iff installed else identity; unknown → `BackendUnavailable`), plus an entry in `backend_status()` and `__all__`.
- An optional **cross-encoder upgrade** (e.g. a `sentence-transformers` CrossEncoder such as `cross-encoder/ms-marco-MiniLM-L-6-v2`) behind `is_available()` (guards on `importlib.util.find_spec("sentence_transformers")` + `"torch"`, like `TransformersNLI` at `/Users/connor/projects/memovox/src/memovox/backends/nli.py:72-77`). Lazy model load + class-level pipe cache, same pattern as `TransformersNLI._pipe_cache`.
- A `rerank_backend: str = "auto"` field on `Settings` (`/Users/connor/projects/memovox/src/memovox/config.py:31-40`), so it is env-overridable (`MEMOVOX_RERANK_BACKEND`) and config-file overridable like every other backend slot.
- The rerank call wired into `augur.answer.ask` **after** `retrieve(...)` and **before** `moment_ids = [...]` (`/Users/connor/projects/memovox/src/memovox/augur/answer.py:87-94`), reordering the `fused` list of `(moment_id, score)` tuples. Reranking happens on the fused candidate set (not per-leg) so the §5 fusion contract is untouched.
- Reranker construction threaded through the `Memovox.ask` facade (`/Users/connor/projects/memovox/src/memovox/sdk.py:63-69`), alongside the existing `get_embedder`/`get_llm` calls.

Lands **ungated** per the thin-fixture discipline; gated only once ≥3 stable golden items exist.

Non-goals / deferrals:
- **No cross-encoder in the CI gate.** The optional upgrade is opt-in, exactly like `deberta-nli`/`sentence-transformers`/`ollama` — it never runs in `make test` or the gated harness.
- **No new retrieval legs.** Visual retrieval (M1.1) and the agentic planner (M2.2) are separate tracks; rerank consumes whatever fused candidate set those produce.
- **No clip stitching** (M2.3) — rerank only reorders moment ids; it does not merge spans.
- **No reranking inside `retrieve()`** — `retrieve` keeps returning the RRF-fused list; reranking is a distinct stage in `ask` (and `synthesize` is explicitly out of scope unless trivially shared — see Open questions).

## Free-path guarantee
- **Defaults OFF:** `rerank_backend="auto"` resolves to the **identity/non-regressing free reranker** on any machine without `sentence-transformers`+`torch` (the CI/test/eval environment), exactly as `nli_backend="auto"` resolves to `LexicalNLI`. The cross-encoder is opt-in and never selected by `make test` or the gated harness.
- **The fallback is a no-op-equivalent reorder.** It returns the **same `(moment_id, score)` set** with **no additions or removals**. If the chosen free impl is identity, the output is the input list unchanged (trivially byte-identical). If the chosen free impl is the lexical/cross-overlap reorder, it must be **stable-sorted** and **proven on golden to not change `mrr`/`ndcg`/`hit_rate`/`groundedness`** before it is allowed to be the default — otherwise it ships as identity and the overlap reorder is registered only as an *opt-in* name. Either way the W2 verify step asserts byte-identical eval output.
- **Order contract preserved:** because `answer.ask` derives `[n]` citation indices and the temporal re-index from the post-rerank `fused` order, the free reranker keeping today's order means **every citation index, deep link, and snippet is byte-identical**, so groundedness and synthesis gates cannot move.
- **Must stay byte-identical:** the `retrieval` block (`hit_rate`/`mrr`/`ndcg`), `groundedness`, `contradiction`, and `synthesis.groundedness` numbers in `python -m eval.harness`, and all 247 `make test` cases.
- **Snapshot pinned:** `rerank_backend="none"` is added to `_FREE_BACKENDS` and the frozen eval-settings snapshot so the determinism guarantee survives a future default flip.

## Eval gate

- **Lands ungated, gated later.** Per the thin-fixture discipline (a 2-talk golden corpus is too small for a stable rerank-lift signal), the `rerank` block is **not** added to `_check_thresholds`. It is promoted to a gate only once **≥3 stable golden items** exist that exercise a reorder-sensitive query.
- **The guard that ships now:** an *equivalence* assertion in the test suite — `rerank.mrr == retrieval.mrr` and `rerank.ndcg == retrieval.ndcg` under the free default (off==today). This is the regression guard: any change that makes the default reranker perturb retrieval order fails immediately.
- **Future gate threshold (when promoted):** rerank `mrr`/`ndcg` ≥ the no-rerank baseline on the same corpus (a non-regression gate: the reranker may improve, never degrade, retrieval). The opt-in cross-encoder is never in the gate.
- **Existing gates stay green and byte-identical:** `retrieval.hit_rate ≥ 0.6`, `groundedness ≥ 0.8`, `contradiction.f1 ≥ 0.5`, `synthesis.groundedness ≥ 0.8` (`harness.py:642-649`). W2's verify step captures the harness JSON before and after to prove byte-identity.

## Risks & mitigations
- **Silent gate drift if the free default reorders.** *Mitigation:* default to identity (or a *proven*-non-regressing stable overlap reorder); W2/W4 assert byte-identical harness output and citation order; if the overlap reorder ever moves a number it ships as opt-in only, not the default.
- **Citation-index / provenance drift (provenance is sacred).** Reranking changes the order `[n]` markers key off (`answer.py:100,127-139`). *Mitigation:* rerank operates on the fused `(id, score)` list *before* citation construction, so `[n]` always re-derives contiguously 1..N over the final order; the temporal re-index still runs after (its sort key is `published_at`, independent of rerank). Test asserts contiguous indices.
- **Score semantics confusion.** The cross-encoder emits a relevance score on a different scale than RRF; the `confidence` shown in provenance is `min(1.0, score*30)` (`answer.py:108`). *Mitigation:* the reranker returns reordered `(id, score)` where score is the rerank score; decide in W2/W3 whether to (a) keep RRF score for confidence display and use rerank score only for ordering, or (b) renormalize — default to (a) to avoid changing the displayed confidence on the free path. See Open questions.
- **Coordination collision with M2.2/M2.3.** All three reshape `ask`. *Mitigation:* this is the **first** Wave-2 track; the rerank seam is a single insertion at `answer.py:90→94` with a new optional `reranker` kwarg — additive and keyword-only, so M2.2's planner-driven retrieve and M2.3's clip stitching layer on without conflict. Build order rerank→planner→clips is honored.
- **Optional dep availability flakiness.** *Mitigation:* `is_available()` guards `find_spec` only (no import side effects), exactly like `TransformersNLI`; the model is lazy-loaded and class-cached; CI only ever exercises the unavailable path.
- **`synthesize` not reranked → inconsistency.** Synthesis (`sdk.py:86+`) uses a different retrieval path. *Mitigation:* explicitly out of scope here (Non-goals); revisit only if W2 shows the reorder helper is trivially shareable. Documented as an Open question.
