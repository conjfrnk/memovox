# M3.4 — Backend A/B benchmark harness

> **Wave:** 3 · **Effort:** M · **Status:** not started
> **Depends on:** M2.1 (rerank slot), M1.1 (Surya/Qwen visual slots + the `visual_result` injection seam) — so there are real upgrade slots to rank · **Owns (single-owner concerns):** none · **Blocks:** none
> **Spec:** §7 ("any backend is A/B-benchmarkable against another"; named backends), §10 (eval dimensions / the measuring stick)

## Goal
Make backend choice **measured, not assumed**. Today the eval harness hardcodes the free stack at exactly two pin-points — `Memovox(store=..., **_FREE_BACKENDS)` in `_ingest_golden` (`/Users/connor/projects/memovox/eval/harness.py:232`) and `get_nli("lexical", config=...)` in `_compute_report` (`/Users/connor/projects/memovox/eval/harness.py:594`) — so there is no way to run the *same* golden metrics under a different embedder / NLI / reranker / visual backend and compare. This track refactors those two pin-points into one **injected `BackendConfig`**, adds an `available_configs()` enumerator that auto-shrinks to a single FREE row on a bare machine (the only row CI ever gates), and emits a **deterministic ranking table** via `python -m eval.harness --benchmark` / `--benchmark --json` / `--benchmark --assert-no-regression`. The free row's metrics are **identical** to `run_eval()`, and two `--benchmark` runs are byte-identical (determinism). Honest scoping: the text-VTT golden corpus can rank embedder/NLI/rerank, but **not** VLM/OCR/visual — those rows light up only once the M1.1 visual subset and `visual_result=` seam exist.

## Why it matters
Spec §7 promises that "any backend is A/B-benchmarkable against another" (`/Users/connor/projects/memovox/src/memovox/backends/base.py:3-4`) and the whole point of the swappable-backend architecture is that you can *prove* an upgrade earns its dependency. By Wave 3 there are finally real upgrade slots to rank: the optional embedder (`sentence-transformers`, `/Users/connor/projects/memovox/src/memovox/backends/embed.py`), the optional NLI (`deberta-nli`, `/Users/connor/projects/memovox/src/memovox/backends/nli.py`), the new `Reranker` slot (M2.1), and the new visual/OCR/VLM slots (M1.1: `VisualEmbedder`, Surya OCR, Qwen2.5-VL). This track turns "we believe the cross-encoder/bge/deberta upgrade helps" into a reproducible ranking table over the golden metrics, while guaranteeing the free default never silently regresses (`--assert-no-regression` is a CI-able gate on the FREE row). It is the spec's "dashboards" Phase-4 bullet, on the measurement side: a single command that says *which* backend config wins on *which* metric.

## Scope (reconciled)
In scope:
- **A `BackendConfig` value type** (frozen dataclass): a stable `name` (e.g. `"free"`, `"st+deberta"`, `"free+cross-encoder"`) plus the full set of backend slot overrides it pins (`embed_backend`, `nli_backend`, `rerank_backend`, `asr_backend`, `llm_backend`, `vlm_backend`, `ocr_backend`, `entity_backend`, and the M1.1 `visual_embed_backend`). The FREE config equals today's `_FREE_BACKENDS` (`/Users/connor/projects/memovox/eval/harness.py:62-70`) plus the new default-OFF flags (`rerank_backend="none"`, `visual_embed_backend="signature"`).
- **Refactor the two hardcoded pin-points into injected backends.** (1) `_ingest_golden` builds `Memovox(store=..., **config.backend_kwargs())` instead of `**_FREE_BACKENDS` (`/Users/connor/projects/memovox/eval/harness.py:232`). (2) `_compute_report` builds the report-time scorer from the config's `nli_backend` instead of the literal `get_nli("lexical", ...)` (`/Users/connor/projects/memovox/eval/harness.py:594`) — the **only** two slots that affect the gated metrics today, so they are the minimum honest refactor. `_FREE_BACKENDS` is kept as the FREE config's source of truth (it stays the frozen snapshot).
- **`available_configs()` enumerator** that returns the list of `BackendConfig`s actually runnable on this machine: it **always** includes the FREE row, and conditionally appends an upgrade row only when **every** non-free slot it pins reports `is_available() == True` (reusing `backend_status()`, `/Users/connor/projects/memovox/src/memovox/backends/__init__.py:168-205`, and `get_reranker`/`get_visual_embedder` availability from M2.1/M1.1). On a bare CI machine this **auto-shrinks to the single FREE row** — the only row CI gates.
- **`run_benchmark(golden_dir, configs=None)`**: runs the *same* `run_eval()` metric path once per available config, returning an ordered list of `(config_name, report)` (FREE first, then upgrade rows in a deterministic, name-sorted order). The free row's report is **metric-identical** to bare `run_eval()`.
- **A deterministic ranking table.** `--benchmark` pretty-prints a table: one row per config, columns for the rankable metrics (`hit_rate`, `mrr`, `ndcg`, `groundedness`, `contradiction.f1`, `synthesis.groundedness`), with a per-metric best marked deterministically (stable tie-break by config name). `--benchmark --json` emits the same data as machine-readable JSON. Output ordering is fully deterministic so two runs diff-clean.
- **`--benchmark --assert-no-regression`**: a CI-able gate that asserts the **FREE row** still clears the four existing thresholds (the same `_check_thresholds` set) — i.e. the benchmark mode can stand in for `--assert-thresholds` on the gated row without inventing new gated numbers. (Cross-config "row X beats row Y" comparisons are **reported, never gated** — upgrade rows are opt-in and machine-dependent.)
- **Eval-settings snapshot extension** (the global discipline): the FREE `BackendConfig` is the single frozen pin of the growing default-OFF surface — it must list every slot, including `rerank_backend="none"` (M2.1) and `visual_embed_backend="signature"` (M1.1), not just the original `_FREE_BACKENDS` keys. A test asserts the FREE config covers **every** `*_backend` field on `Settings` so a future-added slot cannot silently escape the snapshot.

Non-goals / deferrals:
- **No VLM/OCR/visual rows on the text corpus.** The golden VTT fixtures ingest transcript-only (`tessera.run` returns `available=False`, so `visual_vectors` stays empty — see `/Users/connor/projects/memovox/docs/roadmap/m1-1-visual-retrieval.md:49`). VLM/OCR/visual configs therefore *cannot* be ranked here until M1.1's **visual golden subset** + `visual_result=` injection seam exist. The harness must **declare** those rows unrankable on the text corpus (skip with an explicit reason, never silently score 0.0), and become rankable automatically once the visual subset is present.
- **No new gated thresholds.** The benchmark introduces no new gate number. The only gate it can enforce is the FREE row clearing the *existing* four thresholds (`--assert-no-regression`). Cross-config rankings are informational, per the thin-fixture discipline (a 2-talk corpus is too small to gate "config A > config B").
- **No backend implementations.** The embedder/NLI/reranker/visual upgrades are owned by their tracks (embed/NLI exist; rerank = M2.1; visual = M1.1). M3.4 only *ranks* what those tracks provide; it adds no model code.
- **No new optional dependency.** The harness stays pure stdlib; upgrade rows simply don't appear when their deps are absent.
- **No timing/throughput numbers in the gate.** Wall-time per config may be reported (informational), but latency magnitudes are never thresholded — that discipline mirrors M0.1's observability gate (`wall_ms` not thresholded).

## Free-path guarantee
- **Defaults OFF / what stays byte-identical:** On any machine where the optional deps (`sentence-transformers`, `torch`, the cross-encoder, ColPali/Surya/Qwen) are absent — i.e. CI — `available_configs()` returns **exactly one row: FREE**. `run_benchmark` over that single row produces a report **metric-identical** to bare `run_eval()`, asserted by a test that diffs the two dicts key-for-key. So `--benchmark --assert-no-regression` in CI is just `--assert-thresholds` on the FREE row by another name.
- **The FREE `BackendConfig` IS the frozen snapshot.** It pins every `*_backend` slot to its free value (today's `_FREE_BACKENDS` + `rerank_backend="none"` + `visual_embed_backend="signature"`). `_FREE_BACKENDS` remains the literal source of truth the FREE config is built from, so the existing W0.3 hermeticity is preserved unchanged. A test asserts the FREE config names **every** `*_backend` field on `Settings`, so a future slot addition that forgets the snapshot fails CI (this is the global discipline (b) made executable).
- **Two-runs-identical determinism:** because each config runs in its own fresh temp store with offline env pinned and a stable config ordering (name-sorted), two consecutive `--benchmark --json` invocations are byte-identical. A test runs `run_benchmark()` twice and asserts equality.
- **No new dependency, no new gate number.** The harness stays stdlib; the only enforceable gate is the *existing* four thresholds on the FREE row. Upgrade rows never enter any gate.
- **Unrankable rows fail loud, not silent.** VLM/OCR/visual configs on a text-only corpus are **skipped with an explicit "unrankable: no visual subset" reason**, never scored as a misleading 0.0 — preserving "provenance is sacred" for the benchmark table itself.

- **The one enforceable gate is `--benchmark --assert-no-regression` on the FREE row:** it runs the **existing** `_check_thresholds` (`hit_rate ≥ 0.6`, `groundedness ≥ 0.8`, `contradiction.f1 ≥ 0.5`, `synthesis.groundedness ≥ 0.8`) against the FREE config only. This introduces no new threshold — it reuses the four that already gate `main`.
- **The regression guard that ships in-suite:** `run_eval() == run_eval(config=FREE_CONFIG)` and `run_benchmark()[free] == run_eval()` (free row metric-identical), plus two-runs-identical determinism. Any change that makes the FREE-path benchmark diverge from `run_eval()` fails immediately.

## Risks & mitigations
- **Refactor silently moves a gate number** (the two pin-points are exactly the slots that drive the gated metrics). *Mitigation:* W2's `run_eval() == run_eval(config=FREE_CONFIG)` byte-identity test + before/after harness-JSON capture; the FREE config is built literally from `_FREE_BACKENDS` so it cannot drift.
- **Determinism erosion from the growing default-OFF surface** (the review's #2 risk). *Mitigation:* the FREE `BackendConfig` is the single frozen snapshot, and the W1 snapshot-coverage test asserts it names **every** `*_backend` field on `Settings` — a future slot that forgets the snapshot fails CI rather than silently shifting a number.
